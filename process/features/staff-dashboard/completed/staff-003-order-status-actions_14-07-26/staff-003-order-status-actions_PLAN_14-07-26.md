## Public Contracts

### `PATCH /api/staff/orders/:orderId`
- **Auth:** router-level `requireStaff` (inherited) + per-request `resolveBranchScope`.
- **Request body:** `{ status: OrderStatus }`. An optional `etaMinutes` field MAY be present but is
  IGNORED (SPEC Constraint #4).
- **Validation:** zod-parse body (`status` required, must be a valid `OrderStatus`); order-id must be
  a UUID.
- **State machine:** `canTransition(currentStatus, targetStatus)` lookup — invalid → 409.
- **Side effects by target:**
  - `accepted`: `accepted_at = now()`, `estimated_ready_at = placed_at? or now() + branch.estimated_prep_minutes * 60_000`. Decision: use `now()` as the base (ETA is "ready this many minutes from acceptance") — matches SPEC AC-6 wording; test tolerance ±5s.
  - `ready`: `ready_at = now()`.
  - `completed`: `completed_at = now()`; call `creditStarsForOrder(order)` (STAR-001 stub); call `notifyCustomer(order, 'completed')` (PUSH-002 stub).
  - `rejected`: call `notifyCustomer(order, 'rejected')` (PUSH-002 stub). (No dedicated timestamp column exists for rejected; status alone marks terminal.)
  - `cancelled`: `cancelled_at = now()`; call `notifyCustomer(order, 'cancelled')` (PUSH-002 stub).
  - `preparing` / `flavoring`: status change only (no timestamp column).
- **Responses:** `200 { order: StaffOrderDetail }`; `403` cross-branch/unassigned; `404` not found;
  `409` invalid transition or terminal source; `422`/`400` missing/invalid `status`.
- **Note on 422 vs 400:** SPEC API sketch says `422` for missing/invalid `status`. Implement zod
  failure → `422` to match the SPEC contract exactly.

### `GET /api/staff/orders/completed`
- **Auth:** inherited `requireStaff` + per-request `resolveBranchScope`; unassigned → 403.
- **Query:** `WHERE branch_id = assignedBranchId AND status IN ('completed','cancelled','rejected')`,
  `ORDER BY placed_at DESC`.
- **Response:** `200 { orders: StaffOrderSummary[] }` (reuse `serializeStaffOrderSummary`).

### State machine (public within packages/api)
```ts
// order-state-machine.ts (pure, no DB)
const TRANSITIONS: Record<OrderStatus, ReadonlySet<OrderStatus>>
export function canTransition(from: OrderStatus, to: OrderStatus): boolean
export function isTerminal(status: OrderStatus): boolean
```
- `pending → {accepted, rejected, cancelled}`
- `accepted → {preparing, cancelled}`
- `preparing → {flavoring, cancelled}`
- `flavoring → {ready, cancelled}`
- `ready → {completed, cancelled}`
- `completed / cancelled / rejected → {}` (terminal)

## Risk Section

**Risk class:** HIGH — trust boundary (server-side order mutation + branch isolation).
**`mustStopBeforeFinalize: true`** — a manual-first risk-evidence handoff is required before this work
is treated as finalize-ready (auth/permission trust-boundary class per orchestration.md
§High-Risk Execution Handoff).

**Risk predictions (pre-implementation):**
- **Route ordering trap:** `/orders/completed` after `/orders/:orderId` would be swallowed by the
  param route → 404/UUID-parse-fail. Mitigation: Section D step 12 orders it first. This is the
  single most likely silent bug.
- **Enum widening breakage:** `OrderStatus` widening can break exhaustive consumers. Mitigation:
  Section A step 4 grep + `tsc` gate.
- **Migration txn constraint:** `ALTER TYPE ADD VALUE` can't run in a txn; drizzle-kit usually emits
  it standalone but verify (Section A step 3).
- **ETA base ambiguity:** SPEC AC-6 says `placed_at + 15min` in the test but the accept happens later.
  Decision locked: base = `now()` at accept time (the ETA is "ready N minutes from acceptance"); test
  asserts `~now() + prep` with ±5s tolerance. If STAFF/product wants `placed_at`-based, that is a
  one-line change — flagged for VALIDATE review.
- **409 detection on mobile:** the fetch wrapper must surface the HTTP status so the hook can show the
  right inline message. Mitigation: `patchStaffOrderStatus` throws an Error carrying `res.status`.

**Edge cases (high-risk items):**
- Double-tap / concurrent transition: second PATCH hits a now-invalid source status → 409 (no
  optimistic lock needed; the state machine is the guard). Covered by AC-4 re-PATCH-terminal → 409.
- Unassigned staff PATCH → 403 (resolveBranchScope null).
- PATCH with a valid-but-illegal target (e.g. `pending → ready`) → 409. Covered by AC-2.
